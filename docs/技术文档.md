## 1. 项目概述

**TaskConductor** 是一个 AI 驱动的任务流水线编排与 Claude Code 实时观测系统。

### 1.1 核心定位

| 能力层 | 定位 | 当前状态 |
|--------|------|----------|
| **Claude 观测层** | 监听任意 Claude 会话的工具调用，实时展示执行过程 | ✅ 完整实现 |
| **任务流水线层** | 将软件开发任务拆分为 10 个阶段，自动调度 Claude Code 执行 | ⚠️ analysis 阶段完整，其余待补充 |

### 1.2 技术栈

**后端**
- Python 3.11+
- FastAPI（异步 HTTP + WebSocket）
- SQLAlchemy 2.0 Mapped style（ORM）
- SQLite（嵌入式数据库）
- psutil（系统资源采集）
- uvicorn（ASGI 服务器）

**前端**
- React 18 + TypeScript
- Vite（构建工具）
- Tailwind CSS v4
- @xyflow/react（流程图）
- Lucide React（图标）

### 1.3 服务端口

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端开发服务器 | 7070 | Vite dev server |
| 后端 API | 8765 | FastAPI + uvicorn |
| Swagger 文档 | 8765/docs | 自动生成的交互式 API 文档 |
| SQLite 数据库 | 文件 | `backend/task_conductor.db` |

## 2. 整体架构与数据流

### 2.1 两条主要数据流

```
──────────────────────────────────────────────────────────────────
[观测层数据流] 被动监听，任意 Claude 会话均可触发

  用户运行 Claude Code（交互式 / headless）
       ↓
  Claude Code 触发 Hook（9 种事件类型）
       ↓
  ~/.claude/hooks/tc-hook.sh（2s 超时，静默失败）
       ↓
  POST /hooks/claude（main.py）
       ↓
  ┌──────────────────────────────────────┐
  │  1. upsert ClaudeSession（按 session_id）│
  │  2. insert ClaudeEvent               │
  │  3. 记录 MetricsStore（性能指标）    │
  └──────────────────────────────────────┘
       ↓
  WebSocket 双频道广播
  ├→ channel "session:{session_id}" → SessionDetail 页面
  └→ channel "sessions"             → ClaudeMonitorPanel / Sessions 列表

──────────────────────────────────────────────────────────────────
[流水线数据流] 主动驱动，由 API 触发

  POST /api/pipeline/{task_id}/run-analysis（HTTP）
       ↓
  PipelineEngine.next_stage()（状态机校验）
       ↓
  ClaudePool.run(task_id, prompt, cwd)（异步子进程）
       ↓
  子进程：claude -p "..." --output-format stream-json
       ↓
  stream.py 解析 JSON 行 → 流式写入日志文件
       ↓
  WebSocket /ws/task/{task_id} → 前端 LogStream 实时展示
       ↓
  解析结构化输出 → 写入 StageArtifact
       ↓
  Task.stage = "analysis"，Task.status = "pending_approval"
       ↓
  等待 POST /api/tasks/{id}/approve → POST /api/tasks/{id}/advance
```

### 2.2 Hook 安装原理

```
install-hooks.sh
    ↓
写入 ~/.claude/hooks/tc-hook.sh
    ↓
修改 ~/.claude/settings.json，注册 9 个事件：
{
  "PreToolUse":         [{"matcher":"","hooks":[{"type":"command","command":"tc-hook.sh","timeout":5}]}],
  "PostToolUse":        [...],
  "PostToolUseFailure": [...],
  "Stop":               [...],
  "SessionStart":       [...],
  "SessionEnd":         [...],
  "Notification":       [...],
  "SubagentStart":      [...],
  "SubagentStop":       [...]
}
```

**关键细节**：
- `tc-hook.sh` 从 stdin 读取 JSON payload（Claude Code 规范）
- curl 请求加 `--noproxy "*"` 确保 localhost 直连，绕过系统代理
- `|| true` 保证 hook 脚本永远以 0 退出，不阻塞 Claude Code 主流程
- 超时 `--max-time 2` 限制网络等待时间

### 3.2 数据模型（ER 关系）

```
Project (1) ──── (N) Task
                      │
               ┌──────┴──────┐
               │             │
        StageArtifact    ClaudeInstance
           (阶段产物)     (子进程记录)

Project (1) ──── (N) ProjectKnowledge
                      (错误/经验知识库)

ClaudeSession (1) ──── (N) ClaudeEvent
   ↑                          (Hook 事件流)
   └─── (optional FK) Task
```

**各模型字段速查**：

**Task 核心字段**
- `stage`：当前阶段 — input / analysis / prd / ui / plan / dev / test / deploy / monitor / done
- `status`：当前状态 — pending / running / pending_approval / approved / rejected / done / error
- `depends_on`：JSON 序列化的依赖任务 ID 列表
- `worktree_path`：git worktree 隔离路径（dev 阶段使用）
- `branch_name`：对应的 git 分支名

**ClaudeSession 核心字段**
- `session_id`：Claude Code 内部生成的 UUID
- `status`：active / idle / stopped
- `last_seen_at`：最后一条 Hook 事件时间（用于活跃度判断）
- `event_count`：累计事件数（虚拟字段，由关联计数）

### 3.3 流水线状态机

```
阶段顺序（STAGE_ORDER）：
input → analysis → prd → ui → plan → dev → test → deploy → monitor → done

需要审批的阶段（APPROVAL_REQUIRED）：
analysis, prd, ui, plan, test, deploy

转移逻辑：
  需审批阶段：status 必须为 "approved" 才能 advance
  不需审批阶段：status 变 "done" 后自动 advance
```

审批 API 流程：
```
POST /api/tasks/{id}/approve  body: {action: "approve"|"reject", reason: "..."}
    ↓ Task.status = "approved" | "rejected"

POST /api/tasks/{id}/advance
    ↓ PipelineEngine.can_proceed() 校验
    ↓ Task.stage = next_stage
    ↓ Task.status = "pending"（等待下一阶段触发）
```

### 3.4 ClaudePool — headless 子进程管理

```python
# 单例模式
pool = ClaudePool()

# 运行一个 headless Claude 任务
await pool.run(
    task_id=123,
    prompt="分析这个需求：...",
    cwd="/path/to/project",
    log_file="/logs/task_123.log"
)

# 内部实现
subprocess: claude -p "{prompt}" --output-format stream-json
    ↓
stream.py 逐行解析 JSON
    ↓
写入 log_file（前端通过 WS 读取）
    ↓
最终 assistant 消息 → 解析结构化输出 → StageArtifact
```

**关键机制**：
- `_processes: dict[int, asyncio.subprocess.Process]` 跟踪活跃进程
- `kill(task_id)` 可随时中止某个任务的子进程
- 进程退出后自动从 dict 中移除

### 3.5 MetricsStore — 内存性能指标

```python
# 单例，程序重启后清零（非持久化）
store = MetricsStore()

# 记录一次 Claude 调用
store.record_call(
    ttft_ms=342,        # Time To First Token（毫秒）
    duration_s=18.3,    # 总耗时（秒）
    chars=2840,         # 输出字符数
    success=True,
)

# 可查询项
store.summary()         # 调用次数、平均 TTFT、平均耗时、字符速率
store.weekly_stats()    # 过去 7 天每日任务数 + 成功率
store.uptime_pct()      # 服务可用率（基于启动时间）
store.ai_rating()       # 综合评分（审批通过率 × 可用率）
store.token_summary()   # Token 用量（从 ClaudeEvent 中聚合）
store.tool_stats()      # 工具调用次数 Top-N
```

### 3.6 认证机制

```
[本地访问模式]
GET /auth/local → 直接返回 token（仅 localhost 可用，无需 PIN）

[远程访问模式]
启动时 session.py 生成随机 6 位 PIN，打印到终端
POST /auth/pin { pin: "123456" } → { token: "Bearer xxxxx" }

[Token 验证]
所有 /api/* 接口通过 Depends(verify_token) 验证
Token 存储在内存字典，重启后失效
```

### 4.2 状态管理设计

**状态提升原则**：跨页面共享的状态提升到 `App.tsx`，避免各页面重复请求导致数据不同步。

```
App.tsx（顶层状态）
├── authed: boolean              # 是否已认证
├── page: Page                   # 当前页面
├── projects: Project[]          # 全局项目列表（单一来源）
├── activeProjectId: number|null # 当前选中项目
├── activeTaskId: number|null    # 当前选中任务
├── connectionStatus             # 连接状态（用于 UI 反馈）
├── liveEvents: ClaudeHookEvent[]# 全局实时事件（最多 500 条）
└── monitorStatus: WsStatus      # WebSocket 连接状态
```

**页面级状态**：各 Page 组件管理自己的列表、过滤器、弹窗等局部状态。

### 4.3 实时通信

**两种 WebSocket Hook**：

```typescript
// 1. 全局会话监控（Dashboard + Sessions 页面共用）
useClaudeMonitor(onEvent: (e: ClaudeHookEvent) => void)
    连接: WS /ws/sessions
    消息格式: { type: "session_update", data: {...}, ts: "..." }
    适配: 将后端 session_update 转为前端 ClaudeHookEvent 格式

// 2. 单任务日志（TaskPipeline 页面）
useTaskWs(taskId: number, onLog: (line: string) => void)
    连接: WS /ws/task/{taskId}
    消息格式: { type: "log"|"status"|"artifact", data: {...}, ts: "..." }
```

### 4.4 接口管理规范

所有 HTTP 请求统一通过 `lib/api.ts` 发起：

```typescript
// 禁止在组件内散写 fetch，统一调用 api.xxx()
export const api = {
  health: () => request<...>("/health"),
  projects: {
    list: () => request<Project[]>("/api/projects"),
    create: (body) => request<Project>("/api/projects", { method: "POST", ... }),
    tasks: (id) => request<Task[]>(`/api/projects/${id}/tasks`),
  },
  tasks: { get, create, approve, advance, artifacts },
  pipeline: { runAnalysis, runStage },
  metrics: () => request<Metrics>("/api/metrics"),
  system:  () => request<SystemMetrics>("/api/metrics/system"),
  // ...
};
```

**连接配置逻辑**：
- 本地访问（localhost/127.0.0.1）→ 直接连接 `http://localhost:8765`
- 远程访问 → 读取 localStorage 中的 `tc_connection`，使用配置的 tunnelUrl

## 5. 性能指标体系

### 5.1 任务指标

| 指标 | 数据来源 | 说明 |
|------|----------|------|
| 任务总数 | DB `Task` 表 COUNT | 所有项目累计 |
| 按状态分布 | DB `Task.status` GROUP BY | pending / running / done / error 等 |
| 平均完成时长 | `Task.updated_at - Task.created_at` | 仅统计 status="done" 的任务 |
| 审批通过率 | approved / (approved + rejected) × 100% | 反映 AI 输出质量 |

### 5.2 Claude 调用性能指标

| 指标 | 变量名 | 含义 |
|------|--------|------|
| 调用次数 | `call_count` | MetricsStore 累计记录 |
| 首 Token 延迟 | `avg_ttft_ms` | Time To First Token，毫秒 |
| 平均生成耗时 | `avg_duration_s` | 从发起到完成，秒 |
| 字符输出速率 | `avg_chars_per_sec` | 生成速度，字符/秒 |
| 最近 TTFT 序列 | `recent_ttfts_ms[]` | 最近 20 次延迟，用于趋势图 |
| 活跃进程数 | `active_processes` | 当前正在运行的 headless 子进程数 |

### 5.3 Token 与成本指标

| 指标 | 变量名 | 含义 |
|------|--------|------|
| 总输入 Token | `total_input` | 累计 prompt token 数 |
| 总输出 Token | `total_output` | 累计生成 token 数 |
| 缓存写入 | `total_cache_write` | Prompt cache 写入量 |
| 缓存命中 | `total_cache_read` | Prompt cache 读取量 |
| 估算成本 | `total_cost_usd` | 按模型定价估算的 USD 成本 |
| 按模型分布 | `by_model[]` | 各模型调用量和成本 |
| 按小时分布 | `hourly[]` | 24h 内每小时 token 用量 |

### 5.4 会话指标

| 指标 | 数据来源 | 含义 |
|------|----------|------|
| 总会话数 | DB `ClaudeSession` COUNT | 历史累计 |
| 活跃会话数 | `last_seen_at >= (now - 2min) AND status="active"` | 当前在线 |
| 工具调用分布 | DB `ClaudeEvent.tool_name` GROUP BY | 各工具使用频率 |
| 最近工具调用 | 最近 10 条 ClaudeEvent 含 tool_name | 实时监控 |

| 风扇转速 | `sensors.fans[]` | 各风扇当前转速（转/分） |
| 磁盘设备 | `disk_device` | 根分区对应设备名（如 /dev/sda1） |

**系统信息**
| 指标 | 变量名 | 含义 |
|------|--------|------|
| 运行时长 | `uptime_hours` | 系统从启动到现在（小时） |
| 主机名 | `hostname` | `platform.node()` |
| 平台 | `platform` | Linux / Darwin 等 |
| 进程总数 | `process_count` | `len(psutil.pids())` |

### 5.6 KPI 综合指标

| 指标 | 计算逻辑 |
|------|----------|
| AI 评分 | 审批通过率 × 服务可用率 × 100，满分 100 |
| 交互次数 | MetricsStore 中 call_count 累计值 |
| 平均响应时长 | avg_duration_s（Claude 调用平均完成时间） |
| 服务可用率 | (now - start_time) 减去已知宕机时长 / 总时长 |
| 综合可用率 | 与 uptime_pct 相同，用于 Gauge 仪表盘展示 |

## 6. 核心原理深度解析

### 6.1 Hook 事件驱动机制

**Claude Code 的 Hook 规范**：
- Claude Code 在特定生命周期事件发生时，将 JSON payload 写入 stdin，调用注册的 shell 命令
- 9 种事件覆盖了工具执行的完整生命周期（前/后/失败）+ 会话生命周期（开始/结束）+ 通知（等待输入）+ 子 Agent
- `matcher` 字段为空字符串 `""` 表示匹配所有工具，可以设置正则表达式只监听特定工具

**为什么 Hook 比轮询更好**：
- 零延迟：事件发生即推送，无需定时查询
- 低耦合：不侵入 Claude Code 主流程，hook 失败不影响正常使用
- 完整性：覆盖所有工具调用，包括子 Agent 的调用
- 可扩展：任意 Claude Code 会话（包括其他项目）都会被监听

### 6.2 WebSocket 双频道设计

```
为什么需要两个频道而不是一个？

频道 1："sessions"（全局概览）
  - 推送轻量摘要信息（session_id, status, event_count, last_tool）
  - 供 ClaudeMonitorPanel 实时展示所有活跃会话的概览
  - 消费方：Dashboard 右侧面板、Sessions 列表页

频道 2："session:{id}"（单会话详情）
  - 推送该会话的完整事件内容（tool_input, tool_result）
  - 供 SessionDetail 页面展示工具调用的具体参数和结果
  - 只有打开了该会话详情的客户端才订阅此频道

优势：
  - 全局监控时只传输必要的摘要，节省带宽
  - 详情查看时才推送完整内容，避免所有客户端接收无关数据
```

### 6.3 流水线阶段隔离原则

**git worktree 隔离**（dev 阶段规划）：
```
主仓库 repo/
├── .git/
└── src/

worktree：repo/.worktrees/task-{id}/
└── src/  ← 独立文件系统，共享 .git 对象
```

- 多个任务同时开发时，各自在独立的 worktree 中修改文件，互不干扰
- 合并时使用 `git merge --no-ff`，保留完整的变更历史
- worktree 路径存储在 `Task.worktree_path`，进程意外重启后可以恢复

**审批隔离**：
- 每个阶段的输出（StageArtifact）必须经过人工审批才能流转到下一阶段
- 审批记录包含 action（approve/reject）和 reason，形成完整的决策日志
- 拒绝后可以重新触发同一阶段，retry_count 记录重试次数

### 6.4 MetricsStore 设计权衡

**选择内存存储而非 DB 持久化的原因**：
- 实时性要求高：每次 Claude 调用都需要更新指标，DB 写入有延迟
- 指标是瞬态的：TTFT、当前活跃进程数等在重启后本就应该重置
- 避免 DB 膨胀：高频写入会导致 SQLite 文件快速增长
- 简单性：deque(maxlen=500) 自动滚动，无需清理逻辑

**代价**：
- 重启后历史数据清零（用户可以接受，属于"运行时指标"而非"历史报表"）
- 单进程内存，不支持多实例部署（当前不需要）

### 6.5 连接配置与远程访问

```
本地开发：
  前端 → http://localhost:8765 → 后端

远程访问（Cloudflare Tunnel）：
  前端（GitHub Pages）→ https://xxx.trycloudflare.com → 后端

  启动后端时 TC_TUNNEL=1，自动：
  1. 运行 cloudflared tunnel --url http://localhost:8765
  2. 解析输出获取 Tunnel URL
  3. 打印 PIN 码到终端

  用户在前端 Login 页面输入 Tunnel URL + PIN
  → POST /auth/pin 获取 token
  → 存储到 localStorage（tc_connection）
  → 后续请求均带 Bearer token
```

## 7. 风险与思考

### 7.1 安全风险

| 风险 | 级别 | 现状 | 建议 |
|------|------|------|------|
| Token 内存存储 | 中 | 重启后失效，用户需重新认证 | 可加 Redis 持久化或改为 JWT |
| PIN 码强度 | 低 | 6 位数字，可被暴力破解 | 加入失败次数限制（5次锁定） |
| Hook 端点无鉴权 | 中 | `/hooks/claude` 未验证来源 | 可加 HMAC 签名验证 |
| SQLite 并发 | 低 | 多线程写入可能锁冲突 | 开启 WAL 模式（已部分规避） |
| Tunnel URL 泄露 | 中 | URL 可能被日志记录 | 建议定期轮换 Tunnel |

### 7.2 可靠性风险

| 风险 | 场景 | 当前处理 | 改进方向 |
|------|------|----------|----------|
| Hook 事件丢失 | 后端宕机时的 Claude 活动 | tc-hook.sh 静默失败，事件永久丢失 | 本地 buffer + 重试队列 |
| WebSocket 断连 | 网络抖动或后端重启 | 前端无自动重连逻辑 | 实现指数退避重连 |
| 子进程僵尸 | ClaudePool 子进程异常退出 | 有 kill() 方法，但无监控 | 加心跳检测 + 超时自动回收 |
| 大任务 timeout | 复杂任务耗时超过预期 | 无超时机制 | 为各阶段设置最大执行时间 |
| 流水线中断恢复 | 服务重启后 running 任务 | 无恢复机制，任务卡住 | 启动时扫描 running 任务，自动重启或标记 error |

### 7.3 扩展性思考

**当前架构瓶颈**：
- SQLite 单文件：并发写入存在锁竞争，适合单用户场景，多用户需迁移 PostgreSQL
- 单进程内存指标：重启清零，不支持水平扩展
- WebSocket 单节点：不支持多实例部署

**已规划但未实现的功能**：
- `dev` 阶段：需要 Claude Code 交互式模式（非 headless），难点在于如何将会话与任务关联
- 知识库自动提取：当前需手动管理 ProjectKnowledge，应在任务完成后自动提取经验
- 任务依赖调度：`depends_on` 字段已有但未实现等待逻辑
- 并发控制：`Project.max_parallel` 已有但未被 PipelineEngine 消费

### 7.4 观测层 vs 流水线层的设计分歧

**观测层**（已完整）与**流水线层**（部分实现）是两个完全独立的能力，但共用了同一套 ClaudeSession + ClaudeEvent 模型：

```
[困惑点]：
  流水线中 ClaudePool.run() 触发的 headless claude 也会产生 Hook 事件
  → 这些事件会被记录到 ClaudeSession
  → 但如何将这些 session 与对应的 Task 关联？

[当前方案]：
  ClaudeSession.linked_task_id 字段可以手动关联
  但 ClaudePool 启动子进程时，无法提前知道 claude 会分配哪个 session_id

[建议方案]：
  在 headless 调用的 prompt 中注入特定标记（如 task_id）
  在 SessionStart hook 中通过 cwd 或 prompt 特征匹配 Task
```

### 7.5 前端技术选型反思

**正确的决策**：
- 状态提升到 App.tsx：解决了新建项目后 Sidebar 和 Dashboard 不同步的问题
- 集中 api.ts：接口变更只改一处，不会出现各组件各自 fetch 的混乱
- useAutoRefresh hook：统一轮询模式，组件卸载自动清理 interval

**可以改进的地方**：
- Dashboard.tsx 过于庞大（600+ 行）：SystemMetricsPanel 和 ClaudeMetricsPanel 应拆分为独立文件
- 没有全局状态管理库（Zustand/Jotai）：目前靠 props drilling，层级加深后会变复杂
- WebSocket 重连逻辑缺失：网络断开后需手动刷新页面才能恢复
